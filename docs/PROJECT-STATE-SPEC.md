## 1. Purpose and Design Philosophy

### What project-state is

The project-state facility (`.project-state/`) is the operational substrate for any multi-stakeholder project. It stores the authoritative record of formal project structure: milestones, decisions, risks, people, documents, changes, IP disclosures, publications, lessons learned, reports, and communications. Every fact about the project has exactly one home here. Every report and artifact generated by the `project-*` skill suite is derived from this substrate, not authored alongside it.

### The source-of-truth principle

State is the source of truth. Reports are generated from state. State is never updated from reports.

When two artifacts disagree — a slide deck says a milestone is 60% complete but `milestones/M01-data-pipeline-v1.yaml` says 45% — the YAML wins and the slide is regenerated. This is not a convention; it is the enforced operating mode of every skill in the suite. Skills write to the substrate and read from it. They do not read email, slides, or meeting notes as inputs to state updates. Only explicit write operations via `project-state` (the memory layer) change the substrate.

### Why a filesystem, not a database

The substrate is plain YAML, JSON, NDJSON, and Markdown files on a shared drive. There is no server, no API, no credentials, and no subscription. Any sync'd filesystem — Google Drive, OneDrive, Dropbox, a git repository — is a valid backend.

This choice has three consequences:

- **Auditability.** Every file has frontmatter timestamps, every log is append-only, and the entire history is greppable by any team member or auditor without tooling.
- **Portability.** The substrate can be moved to a different drive or dropped into a git repo at any time. YAML + NDJSON + Markdown diff cleanly.
- **No single point of failure.** No API to deprecate, no SaaS to sunset, no secret to rotate. The project's memory persists as long as the files do.

### The review-not-author discipline

Every external-facing artifact passes through human review before it leaves the system:

- Gmail items are always drafts. The `project-notifier` skill has no `send` action. Only `draft`.
- Calendar items are proposed holds or invites. The Project Lead sends them.
- Quarterly claims, SC packs, and public documents are drafts requiring PL sign-off before submission.
- Blog posts respect MPA or equivalent publication review windows before going public.

The system reconstructs status, drafts artifacts, and routes them to the right surface. Humans hold the send button.

### Relationship to the other three state facilities

The project-state facility is the third layer in a four-layer intelligence stack:

| Layer | Facility | Concern |
|-------|----------|---------|
| 1 | Work state (`~/work-state/`) | Evidence — what happened per surface (commits, messages, posts) |
| 2 | Docket state (`~/docket-state/`) | Accountability — informal obligations and pending matters |
| 3 | Project state (`.project-state/`) | Operational structure — formal project management (this facility) |
| 4 | Strategic state (`~/.strategic-state/`) | Direction — thesis, force map, and strategic bets |

**Work state (peer):** Work-state harvests activity evidence from external surfaces (GitHub, Slack, Gmail, GDocs, scsiwyg). It does not read `.project-state/`. Cross-referencing is one-way: work events can carry `project_id` tags that link them to a project, but project-state does not pull from work-state directly. The Project Lead or `project-harvester` skill bridges the two layers by extracting relevant work-state signals and writing them as intel documents into `documents/inbox/`.

**Docket state (peer):** Docket state captures informal obligations that fall outside formal project structure — a commitment made in conversation, a follow-up not yet formalized. SC meeting action items and informal commitments from project communications can be promoted to docket matters by referencing `context.project_id`. Docket matters resolved with a project outcome can be back-referenced in the project activity log via `append-activity-log`.

**Strategic state (downstream consumer):** Strategic state consumes project outcomes for the force map. When milestones are completed and decisions are recorded in `.project-state/`, the `atomic47-strategic-state` skill harvests milestone completions, decisions, and published outcomes to enrich project entries in `~/.strategic-state/force-map/`. Data flows from project-state to strategic-state, not the reverse.

## 2. Directory Layout

```
.project-state/
├── manifest.yaml              ← stable project identity (name, stakeholders, surfaces, packs)
├── state.json                 ← mutable runtime state (phase, counters, last-updated pointers)
├── reporting-matrix.yaml      ← the v2.0 reporting matrix (what, to whom, when, how, via which skill)
├── SCHEMA.md                  ← entity schemas (authoritative; skills validate against this)
├── CONCURRENCY.md             ← concurrency rules (read before any manual edit)
├── README.md                  ← quick-start reference for new team members
│
├── phases/
│   └── {NN}-{phase-slug}/
│       ├── manifest.yaml      ← phase gates, required artifacts, phase decisions
│       └── artifacts/         ← phase-specific deliverables (e.g., signed agreement, kickoff deck)
│
├── milestones/
│   └── M{NN}-{slug}.yaml      ← one file per milestone; M-number matches Schedule A / SOW numbering
│
├── decisions/
│   └── YYYY-MM-DD-{slug}.yaml ← one file per decision, dated and authored
│
├── risks/
│   └── R-{NN}-{slug}.yaml     ← risk register; one file per risk
│
├── changes/
│   ├── change-log/
│   │   └── YYYY-MM-DD-{slug}.yaml   ← non-material changes (no amendment required)
│   └── change-orders/
│       └── CO-{NN}-{slug}.yaml      ← material changes (require amendment or funder approval)
│
├── people/
│   └── {person-id}.yaml       ← one file per person; slug format: {firstname}-{lastname}-{org-short}
│
├── documents/
│   ├── inbox/                 ← documents awaiting classification by project-document-curator
│   ├── working/               ← classified, under revision; not yet source-of-truth
│   └── published/             ← promoted to source-of-truth; indexed in documents/index.yaml
│
├── ip/
│   └── IP-{NN}-{slug}.yaml    ← IP disclosures; owned by project-ip-tracker
│
├── publications/
│   └── YYYY-MM-DD-{slug}.yaml ← proposed and approved publications/presentations
│
├── reports/
│   ├── weekly/
│   │   └── YYYY-Www.md        ← ISO week format; generated by project-status-reporter
│   ├── sc-meetings/
│   │   └── YYYY-QN-{seq}.yaml ← SC/review meeting records
│   └── claims/
│       └── YYYY-QN.yaml       ← quarterly claim records (PIC or equivalent)
│
├── communications/
│   └── {type}/
│       └── YYYY-MM-DD-{slug}.md  ← PIC updates, consortium updates, meeting minutes, blog drafts
│
├── lessons-learned/
│   └── {id}.yaml              ← continuous capture from project start; owned by project-lessons
│
└── logs/
    ├── activity.ndjson        ← all entity writes and events; append-only; never rewritten
    └── decisions.ndjson       ← redundant decision index for fast query without scanning decisions/
```

Every directory that holds YAML entities follows the file-per-entity convention. There is no `milestones.yaml` — only `milestones/M01-*.yaml`, `milestones/M02-*.yaml`, and so on. This is the primary mechanism for safe concurrent writes on a shared drive.

## 3. manifest.yaml Schema

`manifest.yaml` is the stable identity document for the project. It changes infrequently (new team member, new stakeholder, new pack loaded). It is locked with an advisory lockfile before any write. Full schema with all fields annotated:

```yaml
# ============================================================================
# manifest.yaml — project identity and configuration
# Schema version: 2
# ============================================================================

schema_version: 2
manifest_kind: project

project:
  name: "Ai26.10"                          # short identifier; used in filenames and Slack posts
  long_name: "Applied Intelligence 26.10"  # full human-readable title
  one_liner: |
    One-paragraph plain-English description of what the project does and why.
  kind: grant_consortium                   # generic | grant_consortium | client_engagement |
                                           # startup | open_source | mixed
  start_date: 2026-04-01                   # YYYY-MM-DD
  end_date: 2027-03-31                     # YYYY-MM-DD
  packs_loaded:                            # compliance packs active on this project
    - pic-pcais
  reporting_matrix_path: "reporting-matrix.yaml"   # path relative to .project-state/

# ----------------------------------------------------------------------------
# Stakeholders — first-class in v2.0
#
# Every external party that receives reports, attends meetings, or must approve
# artifacts. Stable IDs are referenced by the reporting matrix.
# ----------------------------------------------------------------------------

stakeholders:
  - id: internal.team
    organization: "Atomic47 Labs Inc."
    role: "Project Team"

  - id: funder.pic
    organization: "Protein Industries Canada"
    role: "Funder"
    contacts:
      pm:
        name: "Jane Smith"
        title: "Program Manager"
        email: "jane.smith@pic.global"
      financial_analyst:
        name: "Bob Lee"
        title: "Financial Analyst"
        email: "bob.lee@pic.global"
      director_ip:
        name: "Carol King"
        title: "Director of IP"
        email: "carol.king@pic.global"

  - id: consortium.all
    role: "Consortium Members"
    members:
      - org: "Atomic47 Labs Inc."
        city: "Calgary, AB"
        role: "Project Lead"
      - org: "Partner Org 2 Inc."
        city: "Saskatoon, SK"

# ----------------------------------------------------------------------------
# Phases — preset-driven in v2.0
# ----------------------------------------------------------------------------

phases:
  preset: grant-default                    # grant-default | agile-default | waterfall-default |
                                           # client-engagement-default | open-source-default | custom
  custom_path: ~                           # if preset == "custom", relative path to preset yaml
  current_phase: ~                         # managed by project-phase-gate; do not edit directly

# ----------------------------------------------------------------------------

# ----------------------------------------------------------------------------

# ----------------------------------------------------------------------------

# ----------------------------------------------------------------------------

  slack:
    enabled: true
    channels:
      daily_standup: "#ai2610-standup"
      weekly_report: "#ai2610-reports"

      deploys: "#ai2610-deploys"
  gmail:
    enabled: true

    enabled: true
    calendar_id: "primary"
  blog:
    enabled: true
    site_slug: "ai2610"
    publication_review_required: true      # enforces MPA or equivalent review window

    enabled: true

# ----------------------------------------------------------------------------

# ----------------------------------------------------------------------------

  model: "shared-drive"

  created: 2026-04-23T10:00:00Z
```

### Field index

| Field | Type | Notes |
|-------|------|-------|
| `project.name` | string | Short identifier; used in filenames and surface posts |
| `project.long_name` | string | Full human-readable title |
| `project.kind` | enum | `generic`, `grant_consortium`, `client_engagement`, `startup`, `open_source`, `mixed` |
| `project.start_date` | date | YYYY-MM-DD |
| `project.end_date` | date | YYYY-MM-DD |
| `project.packs_loaded` | string[] | Active compliance packs; controls which profiles are valid |
| `project.reporting_matrix_path` | string | Path to reporting matrix, relative to `.project-state/` |
| `stakeholders[].id` | string | Stable ID referenced by reporting matrix entries |
| `stakeholders[].role` | string | Human label for this stakeholder group |

| `team.project_lead` | person-id | References `people/{id}.yaml` |
| `surfaces.gmail.always_draft` | bool | Must always be `true`; system enforces this |
| `funder.*` | namespace | Populated per loaded pack; see pack docs |
| `concurrency.lock_ttl_seconds` | int | Advisory lockfile TTL; default 300 |

## 4. Entity Schemas

All entity YAML files share a common frontmatter block. Full schemas are in `SCHEMA.md`. This section documents the naming convention, key fields, valid status values, and owning skill for each entity type.

### Common frontmatter (every entity YAML)

```yaml
id: M01-data-pipeline-v1        # matches filename minus extension
kind: milestone                  # see entity type table below
created: 2026-04-23T10:15:00Z
created_by: keystone@stonemaps.org
last_modified: 2026-04-23T10:15:00Z
last_modified_by: keystone@stonemaps.org
phase: 03-planning               # phase slug this entity was created in
```

| Entity | Filename convention | `kind` value | Owning skill |
|--------|-------------------|--------------|--------------|
| Milestone | `M{NN}-{slug}.yaml` | `milestone` | `project-milestone-manager` |

| Risk | `R-{NN}-{slug}.yaml` | `risk` | `project-state` (direct) |

| IP disclosure | `IP-{NN}-{slug}.yaml` | `ip-disclosure` | `project-ip-tracker` |

## 5. state.json Schema

`state.json` holds mutable runtime state. It is updated by skills after significant events (phase transition, milestone completion, report generation). It uses advisory lockfile discipline before writes. Skills read this file to answer "where are we now?" without scanning all entity files.

```json
{
  "schema_version": 2,
  "current_phase": "03-planning",
  "project_health": "green",
  "last_validated": "2026-05-05T09:00:00Z",
  "reporting": {
    "last_weekly_report": "2026-W17",
    "last_sc_meeting": "2026-Q1-01",
    "last_claim_submitted": "2026-Q1",
    "next_claim_due": "2026-07-20",
    "next_sc_meeting": "2026-07-10"
  },
  "counters": {
    "milestones_total": 5,
    "milestones_complete": 1,
    "milestones_at_risk": 0,
    "decisions_total": 3,
    "risks_open": 4,
    "changes_total": 2,
    "ip_disclosures": 1
  }
}
```

### Field reference

| Field | Type | Notes |
|-------|------|-------|
| `schema_version` | int | Must be `2` for v2.0 projects |
| `current_phase` | string | Phase slug; set by `project-phase-gate` |
| `project_health` | enum | `green`, `yellow`, `red`; set by `project-status-reporter` |
| `last_validated` | ISO datetime | Last time `project-state validate` ran clean |
| `reporting.last_weekly_report` | ISO week | e.g., `2026-W17` |
| `reporting.last_sc_meeting` | string | e.g., `2026-Q1-01` |
| `reporting.last_claim_submitted` | string | e.g., `2026-Q1` |
| `reporting.next_claim_due` | date | YYYY-MM-DD |
| `reporting.next_sc_meeting` | date | YYYY-MM-DD |
| `counters.*` | int | Maintained as a fast-query cache; recomputable from entity files |

Counters are authoritative only for quick reads. If they drift from entity files, run `project-state validate` to reconcile.

## 6. The Reporting Matrix

### What it is

`.project-state/reporting-matrix.yaml` is the central v2.0 concept. It encodes the answer to: for each stakeholder group this project communicates with, what report do they receive, at what cadence, in what format, by which surface, and which skill produces it?

In v1.x this was implicit and hard-coded for a single funder. The matrix externalizes it as data, which turns the system from a grant-management tool into a generic multi-stakeholder reporting platform of which grants are one instance.

### Schema

```yaml
schema_version: 2
manifest_kind: reporting_matrix

entries:
  - id: <unique-string>               # unique within this file; referenced in logs
    stakeholder_group: <id>           # must match a stakeholder.id in manifest.yaml
    report: <string>                  # human-readable report kind
    cadence: <cadence object>         # see cadence kinds below
    format: <format>                  # md | docx | xlsx | pdf | html | mdx | meeting
    surface: <surface>                # see surface values below
    generator: <project-* skill>      # skill that produces this report
    profile: <pack-profile-id>        # optional; required when generator needs pack config
    description: <string>             # optional; one-paragraph explanation
    optional: <bool>                  # if true, entry is opt-in and not auto-dispatched
```

### Cadence kinds

```yaml
# Weekly
cadence: { kind: weekly, day: monday, lead_time_hours: 8 }

# Bi-weekly
cadence: { kind: bi-weekly, day: friday-odd-week }

# Monthly
cadence: { kind: monthly, day_of_month: 1 }
cadence: { kind: monthly, day_of_month: "last-business-day" }

# Quarterly with fixed annual dates
cadence: { kind: quarterly, days_of_year: ["04-20", "07-20", "10-20", "01-20"], lead_time_days: 14 }

# Quarterly aligned to quarter start
cadence: { kind: quarterly, alignment: "first-month-of-quarter" }

# Annual
cadence: { kind: annual, due_month: "04" }

# Sprint-aligned (agile pack)
cadence: { kind: sprint-aligned, timing: "last-day-of-sprint" }

# Ad-hoc on a substrate event
cadence: { kind: ad-hoc, on_event: "milestone.completed" }

# Post-event within N business days
cadence: { kind: post-event, on: "sc-meeting.held", within_business_days: 5 }

# On-publish when a path changes
cadence: { kind: on-publish, trigger: "documents/published/" }
```

All cadences support optional `lead_time_days` and `lead_time_hours` to control when the draft is produced ahead of the deadline.

### Surface values

| Value | Delivery mechanism |
|-------|--------------------|
| `slack` | Auto-post to configured channel |
| `gmail.draft` | Gmail draft only; never auto-sent |
| `calendar.invite` | Proposed calendar hold + invite |
| `blog` | scsiwyg blog post (publication review enforced) |
| `website` | Project website deployment |
| `github-release` | GitHub release (open-source pack) |
| `github-discussion` | GitHub discussion (open-source pack) |

Surfaces can be combined: `surface: gmail.draft + slack`, `surface: calendar.invite + gmail.draft`.

### Generators

| Generator | Produces |
|-----------|---------|
| `project-status-reporter` | Weekly reports, stakeholder updates, ad-hoc status, dashboard snapshots |
| `project-funder-reporting` | Recurring funder reports per a pack profile (claims, invoices, board packs) |
| `project-review-meeting` | Meeting packs and minutes per a pack profile (SC, board, QBR, retro) |
| `project-blog-publisher` | Narrative blog posts with MPA review window enforcement |
| `project-website-publisher` | Reference docs at stable URLs; auto-deploys on document publish |
| `project-change-register` | Change orders drafted for stakeholder approval |
| `project-ip-tracker` | IP disclosures routed to configured recipient |

### How the orchestrator uses the matrix

`project-orchestrator` runs on a scheduled tick. On each run it:

1. Reads `reporting-matrix.yaml` and `state.json`.
2. For each entry, computes whether it is due based on `cadence` and the last-generated pointer stored in `state.json` or report filenames.
3. Dispatches overdue entries to their named `generator` skill.
4. Generators write artifacts to `reports/` or `communications/`, then call `project-notifier` to route them to the configured surface.
5. Orchestrator updates `state.json` reporting pointers.

### How packs seed the matrix

Each compliance pack ships a `reporting-matrix-defaults.yaml`. When `project-scaffolder seed-matrix` is run (or equivalently, "seed reporting matrix from packs"), the defaults from each loaded pack are merged into the project's matrix. The project team then customizes cadences, removes entries they don't need, and adds project-specific entries not covered by packs.

### Validation rules

| Rule | Error condition |
|------|----------------|
| Unique IDs | Two entries share the same `id` |
| Stakeholder references | `stakeholder_group` is not defined in `manifest.yaml` |
| Generator exists | `generator` is not an installed `project-*` skill |
| Profile bound | `profile` references a profile not shipped by a loaded pack |
| Cadence kind valid | `cadence.kind` is not one of the supported kinds |
| Format valid | `format` is not one of: `md`, `docx`, `xlsx`, `pdf`, `html`, `mdx`, `meeting` |

```yaml
phases:

  - id: "02-approval"
    label: "Submission & Approval"
```

### Gate enforcement

Each phase has `gate_in` and `gate_out` conditions plus an optional `required_artifacts` list. `project-phase-gate` validates that all gate conditions are met before allowing a transition. Gate conditions are checked against:

- Entity counts and statuses in `.project-state/`
- Required artifact presence in `documents/published/`
- Pack profile override conditions (e.g., a funder-specific checklist)

If any gate condition is unmet, the transition is blocked and the skill reports exactly which conditions are outstanding.

### Phase directory

Each phase has a corresponding directory at `phases/{NN}-{slug}/` containing:

- `manifest.yaml` — phase-specific gates, required artifacts, and any decisions recorded during the phase
- `artifacts/` — deliverables specific to this phase (signed agreement, kickoff deck, etc.)

## 8. Operations via project-state

`project-state` is the memory layer. Every other skill calls through it for reads and writes. Direct manual edits to substrate files are permitted only when skills are unavailable and must follow `CONCURRENCY.md`.

### Read operations

| Operation | Description |
|-----------|-------------|
| `get-manifest` | Returns `manifest.yaml` parsed as structured data |
| `get-state` | Returns `state.json` |
| `get-entity kind id` | Returns a single entity file (e.g., `get-entity milestone M01-data-pipeline-v1`) |
| `list-entities kind` | Lists all entity files of a given kind; supports filter flags |
| `list-entities kind --status open` | Filtered list; `--status`, `--owner`, `--phase` are common filters |
| `tail-log [n]` | Returns the last N lines of `logs/activity.ndjson`; default 20 |

### Write operations

| Operation | Description |
|-----------|-------------|
| `create-entity kind fields` | Creates a new entity file; generates `id` from kind + fields; appends creation event to activity log |

## 9. Concurrency Model

The substrate is designed for safe concurrent access on a shared drive. Seven principles govern all reads and writes.

### Principle 1 — file-per-entity

Every entity is a separate file. Two writers working on different milestones, different risks, or different decisions never touch the same file. Monolithic files (`manifest.yaml`, `state.json`, `reporting-matrix.yaml`) are write-rare and handled by Principle 3.

### Principle 2 — frontmatter timestamps

Every writable entity file has `last_modified` and `last_modified_by` fields. Before writing, read the current file. If `last_modified` is newer than the version you started from, you have a conflict and must re-read and reconcile before writing.

### Principle 3 — advisory lockfiles for monolithic files

For `manifest.yaml`, `state.json`, `reporting-matrix.yaml`, and any `tracking/*.xlsx`:

1. Write `<filename>.lock` containing `{"actor": "...", "acquired": "<ISO>", "ttl_seconds": 300}`.
2. If a `<filename>.lock` already exists and `acquired + ttl_seconds` is in the future, wait or abort — never clobber.
3. Complete the edit.
4. Delete the lockfile.

Stale lockfiles (TTL elapsed) may be ignored and overwritten. The default TTL is 300 seconds (5 minutes).

### Principle 4 — append-only logs

`logs/activity.ndjson` and `logs/decisions.ndjson` are append-only. They are never rewritten. Appending a single line under 4KB is atomic on all supported filesystems. To correct a past log entry, append a `log.correction` event; do not rewrite history.

### Principle 5 — deterministic filenames

Filenames are derived deterministically from entity content. For milestones: `M{NN}-{kebab-slug}.yaml` where `NN` is the Schedule A number and the slug comes from the milestone title. Two agents independently creating the same milestone produce the same filename, which resolves to a safe conflict at the filesystem level rather than a silent data split.

### Principle 6 — xlsx trackers are single-writer

Any `.xlsx` under `tracking/` is treated as single-writer. Apply the lockfile discipline from Principle 3 before editing. xlsx files are generated from YAML source-of-truth. If they diverge from YAML, regenerate them.

### Principle 7 — git-friendly

YAML + NDJSON + Markdown diff cleanly. The `.project-state/` directory can be placed in a git repository at any time. Binary docx/xlsx under `documents/` and `tracking/` can be managed with Git LFS.

### Manual edit checklist

If skills are unavailable and you must edit by hand:

1. Check that no `<file>.lock` exists (or that it is expired).
2. Note the current `last_modified` value before you start.
3. Make your edit. Update `last_modified` to the current UTC time and set `last_modified_by` to your email.
4. Append an event to `logs/activity.ndjson` describing what you did.
5. No bulk rewrites. One entity per edit session.

## 10. Document Promotion Pipeline

Documents move through a three-stage pipeline before they can be referenced by other entities or included in external reports.

### Stages

| Stage | Path | Description |
|-------|------|-------------|
| Inbox | `documents/inbox/` | Unclassified documents awaiting review; no entity may reference these |
| Working | `documents/working/` | Classified; being edited; not yet source-of-truth |
| Published | `documents/published/` | Promoted to source-of-truth; indexed; referenceable |

Only `published/` documents can be referenced by milestones, decisions, change orders, IP disclosures, or external reports. Any reference to a `working/` or `inbox/` document in an entity file is a validation error.

### Promotion

`project-document-curator` owns the pipeline. Promotion from `working/` to `published/`:

1. Curator classifies the document (type, visibility tier, owning entity references).
2. Curator writes a record to `documents/index.yaml` with the document ID, path, type, visibility, and promotion timestamp.
3. Curator moves the file to `documents/published/`.
4. Activity log receives a `document.sot.promoted` event.
5. `project-website-publisher` (if enabled and `auto_deploy_on_publish: true`) picks up the event and rebuilds the relevant website pages.

### Visibility tiers

Documents carry a visibility classification that controls which surfaces they can appear on:

| Tier | Visible to | Website behavior |
|------|-----------|-----------------|
| `internal` | Project team only | Not published to website |
| `consortium` | All consortium members | Published with consortium-level access |
| `public` | Anyone | Published publicly; requires SC or funder publication review |

`require_review_for_public: true` (the default) enforces that no document reaches the `public` tier without the MPA publication review window having elapsed.

### documents/index.yaml

`project-document-curator` maintains a machine-readable index at `documents/index.yaml`. Each entry:

```yaml
- id: mpa-signed-2026-04-01
  path: documents/published/mpa-signed.pdf
  kind: governing-agreement
  visibility: consortium
  promoted: 2026-04-23T10:45:00Z
  promoted_by: keystone@stonemaps.org
  title: "Master Project Agreement — Signed"
  referenced_by:
    - kind: milestone
      id: M01-data-pipeline-v1
```

## 11. Relationship to Other State Facilities

### Work state (`~/work-state/`)

Work state is the evidence layer. It harvests activity from external surfaces — GitHub commits, Slack messages, Gmail threads, Google Docs edits, scsiwyg posts — and stores them as structured records keyed by surface and date. Work-state records do not alter `.project-state/` directly.

Cross-referencing is one-way from work events to project IDs. A GitHub commit record in work-state can carry `project_id: ai2610`; the presence of that tag does not write anything to `.project-state/`. The `project-harvester` skill bridges the two layers: it reads work-state records filtered by project ID and writes relevant signals as documents into `documents/inbox/`, where the curator pipeline takes over.

Work-state does not consume project-state. There is no back-reference from work-state into `.project-state/` schemas.

### Docket state (`~/docket-state/`)

Docket state is the accountability layer for informal obligations — a commitment made in a conversation, a follow-up not yet formalized into a project milestone. Project-state and docket-state are peers; neither reads the other's files directly.

Two cross-reference patterns are supported:

- **Project to docket:** SC meeting action items and informal commitments captured in project communications can be promoted to docket matters. The docket matter carries `context.project_id` referencing the project.
- **Docket to project:** When a docket matter is resolved through a project outcome (a milestone completes, a decision is recorded), the resolution can append a `docket.matter.resolved` event to the project activity log referencing the docket matter ID.

Both patterns are manual or skill-mediated. There is no automatic synchronization.

### Strategic state (`~/.strategic-state/`)

Strategic state is a downstream consumer of project-state. It is not a peer. The `atomic47-strategic-state` skill reads `.project-state/` to enrich project entries in the force map:

- Milestone completions → thesis evidence for strategic bets linked to this project
- Decisions recorded → input to strategic decision log
- Published outcomes → signals for the force map update

Strategic state never writes to `.project-state/`. Data flows one way: from project-state to strategic-state, at harvest time.

The strategic-state skill does not modify entity files. It reads `milestones/`, `decisions/`, and `documents/published/` to extract signals. The activity log is not consumed by strategic-state directly; milestone and decision entity files are the primary inputs.
