# Project SR&ED Reviewer

## Purpose

Review SR&ED T661 claim narratives for clarity, eligibility framing, and audit resistance before advisor handoff or CRA submission. Reads from the `sred/` substrate (TU/EX/ADV entities + evidence log generated by `project-sred-tracker`) or accepts pasted T661 draft text directly.

The goal is to lower audit risk and strengthen defensibility without overclaiming. Every improvement must be grounded in the user's own facts — this skill never invents experiments, results, or evidence.

**Not tax advice.** This skill provides submission-strengthening guidance. Eligibility determinations, filing decisions, and claim amounts require a qualified SR&ED consultant or tax advisor.

## Input modes

### Mode A — From substrate (primary)
Pull T661 narrative content from `sred/uncertainties/`, `sred/experiments/`, `sred/advancements/`, and `sred/evidence-log.ndjson` via `project-sred-tracker`. Run full review against the structured entities.

### Mode B — Pasted draft
Accept pasted Section E, F, and/or G text directly. Parse into TU/EX/ADV structure, then apply the same review workflow. Offer to sync findings back to the substrate.

## Review workflow

### Step 1 — Identify the claim structure

Extract or infer:
- List of technological uncertainties claimed
- Experimental streams and work performed
- Technological advancements claimed
- Any cost-related framing

### Step 2 — Section E: Technological Uncertainties

For each TU, test whether it is:
- Framed as unknown **at the outset** (not discovered during development)
- **Technological** — not commercial, operational, workflow, or user-requirements-based
- Specific enough to map to experiments
- Not just standard integration difficulty, architecture complexity, or "we wanted to build"

**Flag risky language:**

| Risky | Why | Preferred alternative |
|-------|-----|----------------------|
| "we needed to build" | Product goal, not technological uncertainty | "It was uncertain whether" |
| "the user required" | Requirements, not research | "No established method existed for" |
| "we wanted to create" | Desire, not knowledge gap | "It was not known whether" |
| "available guidance did not explain how to configure..." | Implementation gap, not uncertainty | "It was uncertain whether the required [property] could be achieved..." |
| "these tools would behave together" | Integration difficulty | "Whether [specific technical property] could be achieved under [constraints]" |

### Step 3 — Section F: Work Performed

For each EX, confirm presence of:
- **Hypothesis** — what was being tested
- **Experimental implementation** — what was built or tried
- **Testing or evaluation** — what was assessed
- **Observations** — what was found, including failures, limitations, variability, iteration

**Flag narratives that read like pure build work:**
- Straightforward module implementation without a hypothesis
- Feature completion language ("the team built", "the team created")
- No mention of failure, edge cases, unexpected constraints, or refinement cycles
- Deployment or polish work presented as experimentation

**Strengthen by restoring the pattern:**
Hypothesis → experiment → result → implication

### Step 4 — Section G: Technological Advancements

For each ADV, verify:
- States **knowledge gained**, not **product completed**
- Is proportionate to the experimental evidence beneath it (don't claim broader knowledge than the results support)
- Maps back to at least one TU and one EX

**Flag weak advancement claims:**

| Risky | Preferred |
|-------|-----------|
| "we built" | "the work established that" |
| "we created" | "the investigation demonstrated that" |
| "the platform supports" | "the work identified the conditions under which" |
| "we delivered" | "this advanced the knowledge base by" |
| Advancement broader than the experiment evidence | Narrow the claim to match what was actually found |

### Step 5 — Cross-section traceability

Build the chain: **uncertainty → experiment → result → advancement**

For each TU:
- Is there at least one EX? If not: flag as incomplete
- Does the EX have a documented result? If not: flag as weak
- Does the EX link to an ADV? If not: flag as missing

For each ADV:
- Does it trace back to a TU? If not: flag as unsupported
- Does the TU → EX → ADV chain tell a coherent story? If not: flag inconsistency

### Step 6 — Cost defensibility (when relevant)

If cost categorization is provided or requested, assess:

| Category | Typical examples |
|----------|-----------------|
| **Eligible** | Feasibility experiments, prototype logic tied to uncertainty, testing/analysis of technical results, iterative refinement to resolve technical unknowns |
| **Partially eligible** | Support work that enables experimentation but serves other purposes too, broad architecture work with experimental and product components |
| **Non-eligible** | UI polish, routine integration, production deployment, user documentation, customer customization, standard debugging unrelated to uncertainty |

Flag any costs claimed as eligible that appear to be routine engineering based on the Section F narrative.

### Step 7 — Quantified results check

If any Section F narrative contains a specific percentage, metric, or quantified result:
- Flag it for verification
- Ask: is there a retained test protocol, source data, date, and version context?
- If not: recommend removing the figure (a specific number without evidence is an audit target)

## Output patterns

### Pattern A — Quick verdict

Use when user asks for overall assessment or doesn't want full detail:

```
Overall assessment: strong / borderline / weak

Biggest strengths:
- ...

Biggest audit risks:
1. [Risk] — [Why] — [What to do]
2. ...

Highest-priority fixes before submission:
- ...
```

### Pattern B — Section-by-section critique

Use when user provides draft text or asks for full review:

```
Section E critique:
  Strengths: ...
  Risks: ...
  Specific edits: ...

Section F critique:
  Strengths: ...
  Risks: ...
  Specific edits: ...

Section G critique:
  Strengths: ...
  Risks: ...
  Specific edits: ...

Cross-section issues: ...
Cost-framing issues: ...
```

### Pattern C — Rewrite pass

When the user asks for improved language:
- Preserve technical substance — never invent experiments or results
- Apply CRA-preferred language patterns from the pack profile
- Strengthen uncertainty, experiment, and advancement framing
- Be explicit when details are missing ("this section needs a specific result before this language is defensible")

### Pattern D — CRA attack simulation

When user asks for audit preparation:

For each likely CRA objection:
```
Reviewer concern: "This looks like standard software development."
Why they might say this: [specific language or structure that triggered it]
Best defense from the current record: [what in the claim supports SR&ED eligibility]
What evidence would strengthen the defense: [what records to retain or cite]
```

Common CRA objections to simulate:
- "This appears to be integration of known tools"
- "Where is the technological uncertainty — this looks like engineering execution"
- "Where is the failure or iteration — this reads like a successful implementation"
- "How is this a technological advancement rather than a product feature?"
- "This uncertainty was resolvable by standard practice"
- "The advancement claim is broader than what the experimental evidence supports"

## Discipline rules

- **Stay grounded in the user's facts.** Never fabricate technical evidence, experimental results, or failed attempts. If details are missing, say what additional evidence would strengthen the claim.
- **Be conservative.** Mark borderline situations as borderline. Do not push weak claims toward eligible.
- **Do not state eligibility as certain.** Frame all output as submission-strengthening guidance.
- **Be concrete.** Vague encouragement is useless. Name specific sentences, specific risks, specific fixes.
- **Proportionality check.** Every advancement should match the scope of the linked experimental evidence. Flag when they don't.

## Standard checklist (used in every review)

1. Is each uncertainty technological and unknown at the outset?
2. Is each uncertainty specific enough to test?
3. Does each work stream describe a hypothesis and an experiment?
4. Does the narrative show iteration, variability, limitations, or failed paths?
5. Does each advancement describe knowledge gained rather than functionality delivered?
6. Can every advancement be traced back to a TU and an EX?
7. Is routine development separated from experimental development?
8. Are there product, business, or user-need statements presented as technological uncertainty?
9. Are there quantified results without documented evidence?
10. Would a skeptical CRA reviewer read this as research-oriented rather than routine software development?
